# Portfolio Demos — Notes, Questions & Concerns

> Address these after the main plan is executed. No blockers to execution.

---

## Questions

### Q1: Demo linking strategy — iframe embed vs. separate URLs?

The plan links from `apps/client` to demos as separate Vite apps (separate ports in dev, separate deploy origins in prod). Embedding via iframe is possible but adds cross-origin complexity. Recommend separate URLs for simplicity and inspectability (interviewers can directly open the demo URL). Revisit if the portfolio needs a unified SPA feel.

### Q2: Shared server vs. per-demo server?

The plan extends `apps/server` to serve the AI pipeline SSE routes. This is fine for dev (single server). In prod, all demos would share the same server process. If demos diverge significantly, consider splitting to `apps/server-ai-pipeline` later. Not a concern for Phase 1.

### Q3: Moon + Turbo coexistence — which to use in CI?

Plan keeps `turbo.json` during migration. Moon's `moon ci` is used in GitHub Actions. Turbo is left in place but not called from CI after Phase 1. Can be fully removed once Moon is verified.

### Q4: Demo ports in dev

- `client`: 3000
- `demo-ai-pipeline`: 3001
- `demo-datavis`: 3002
- `demo-xscan`: 3003
- `server`: 4000

All demos proxy `/api` → 4000. Confirm no conflicts with other local services.

### Q5: Fixture file size

A full 500-token markdown response with Mermaid could be ~3-4KB as a JSON fixture. Five fixtures = ~20KB. Fine to commit to git. If they grow large (>50KB each), consider gzip or moving to a CDN.

## Concerns

### C1: Mermaid + SSR/streaming

Mermaid calls `mermaid.render()` in the browser. This is client-only. If RSC or SSR is ever introduced, Mermaid needs to be wrapped in a client boundary. No SSR planned for this demo — noted for future.

### C2: Shiki bundle size

`shiki` with all grammars is large (~2MB). Use `createHighlighterCore` with only needed language grammars (`typescript`, `javascript`, `bash`, `json`, `yaml`, `css`). Lazy-load the highlighter. Check bundle size after Phase 3.

### C3: `dangerouslySetInnerHTML` usage

Used in two places: `MermaidBlock` (Mermaid-generated SVG) and `CodeBlock` (Shiki-generated HTML). Both are safe because the HTML is generated by the library, not from user or LLM input. `rehype-sanitize` handles the markdown → HTML path. Document this distinction in the component JSDoc.

### C4: Rate limiting on live LLM endpoint

In-memory rate limiter resets on server restart. Fine for a demo. If deployed and traffic spikes, consider Redis-backed rate limiting or simply keeping `STREAM_MODE=fixture` as default in public deployment.

### C5: `ANTHROPIC_API_KEY` in deployment

Never commit the key. Ensure `.env.production` is in `.gitignore` (it is). Use Railway/Fly.io secret injection for deployment. The plan validates `ANTHROPIC_API_KEY` in `@workspace/config` only when `STREAM_MODE=live`.

### C6: Moon `moon init` may conflict with existing `turbo.json` or root scripts

Run `moon init` carefully. Moon will prompt about existing task runner — choose "no" when asked to remove Turbo. Both can coexist via `moon.yml` per-project task definitions that call the same underlying commands (`pnpm exec vite`, `pnpm exec tsc`, etc.).
